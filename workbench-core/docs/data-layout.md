# Workbench Data Layout

Workbench persists everything under a configurable `dataDir` (default `~/.workbench`)
or `$XDG_STATE_HOME/workbench`. The CLI resolves the directory via `internal/config/effectiveConfig()`
and the `--data-dir` flag / `WORKBENCH_DATA_DIR` environment variable.

## Sessions vs. runs

Workbench separates work into sessions and runs. Each session holds a stable goal/context,
while runs capture a single agent execution with its workspace and artifacts.

- **Sessions** live under `<dataDir>/sessions/<sessionId>/`. Each session directory contains:
  - `session.json` – metadata (id, goal, current run, run index).
  - `history/history.jsonl` – session-scoped operation history recorded by `internal/store/history.go`.
- **Runs** live under `<dataDir>/runs/<runId>/`. Each run directory contains:
  - `run.json` – run metadata persisted via `internal/store/run.go`.
  - `events.jsonl` – chronological history of host/agent events.
  - `scratch/` – run-scoped workspace exposed as `/scratch` inside the sandbox.
  - `artifacts/`, `results/`, `log/` – CLI-observable outputs generated by tools or agent actions.
  - `memory/` – per-run memory markdown (`memory.md` plus `update.md` staging files).
  - `profile/` – host-level profile data injected into runs.

Helper functions in `internal/fsutil/paths.go` ensure this layout remains consistent (`GetRunDir`,
`GetArtifactDir`, etc.), so other packages can rely on deterministic paths.

## Agent consumable mounts

Workbench exposes an Agentic File System (AFS) inside each run:

- `/project` – the host working directory (default `.`) mounted via `--workdir` / `WORKBENCH_WORKDIR`.
- `/scratch` – the run-local workspace (maps to `<dataDir>/runs/<runId>/scratch`).
- `/results` – where tool.run writes structured responses: `/results/<callId>/response.json`.
- `/log`, `/memory`, `/profile`, `/tools` – read-only mounts that surface logs, memory proposals,
  global profile state, and discovered tool manifests (`internal/tools`).

Understanding this layout makes it easier to inspect sessions/runs manually (e.g., `cat data/sessions/<id>/session.json`).
All paths are derived via helpers in `internal/fsutil` and validated through `internal/store`.
