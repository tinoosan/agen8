# Workbench Data Layout

Workbench persists everything under a configurable `dataDir` (default `~/.workbench`)
or `$XDG_STATE_HOME/workbench`. The CLI resolves the directory via `internal/config/effectiveConfig()`
and the `--data-dir` flag / `WORKBENCH_DATA_DIR` environment variable.

Core metadata (sessions, runs, events, history) is stored in a SQLite database at:

- `<dataDir>/workbench.db`

## Sessions vs. runs

Workbench separates work into sessions and runs. Each session holds a stable goal/context,
while runs capture a single agent execution with its workspace and artifacts.

- **Sessions** are stored in SQLite (`sessions` table), including the session JSON and run index.
  - Session history (`/history/history.jsonl`) is read from SQLite but still exposed as JSONL via VFS.
- **Runs** are stored in SQLite (`runs` and `events` tables), while run directories remain on disk:
  - `workspace/` тАУ run-scoped workspace exposed as `/workspace` inside the sandbox.
  - `artifacts/`, `log/` тАУ CLI-observable outputs generated by the agent and host tooling.
  - `inbox/`, `outbox/`, `plan/` тАУ task IO + planning workspace for the run.

Shared memory lives outside run directories:

- `memory/` тАУ shared memory (`MEMORY.MD`, plus daily `YYYY-MM-DD-memory.md` files).

Event logs are mirrored to `<dataDir>/agents/<agentId>/log/events.jsonl` so `/log` remains offset-based.

Helper functions in `internal/fsutil/paths.go` ensure this layout remains consistent (`GetRunDir`,
`GetArtifactDir`, etc.), so other packages can rely on deterministic paths.

## Agent consumable mounts

Workbench exposes an Agentic File System (AFS) inside each run:

- `/project` тАУ the host working directory (default `.`) mounted via `--workdir` / `WORKBENCH_WORKDIR`.
- `/workspace` тАУ the agent-local workspace (maps to `<dataDir>/agents/<agentId>/workspace`).
- `/log` тАУ read-only mount that surfaces the run event stream.
- `/skills` тАУ read-only mount that surfaces user-defined skills.
- `/plan` тАУ run-scoped planning workspace (`HEAD.md` + `CHECKLIST.md`).
- `/inbox`, `/outbox` тАУ run-scoped task IO.
- `/memory` тАУ shared agent memory (`MEMORY.MD`, plus daily `YYYY-MM-DD-memory.md` files).

Understanding this layout makes it easier to inspect sessions/runs manually (e.g., querying `workbench.db`
and inspecting `<dataDir>/agents/<id>/log/events.jsonl`). All paths are derived via helpers in
`internal/fsutil` and validated through `internal/store`.

### VFS paths vs. shell cwd

- File operations (`fs.read`, `fs.write`, etc.) use VFS paths like `/project/src/main.go`.
- Shell commands run in the host workdir root by default; `cwd` should be **relative** (e.g., `src/`), not a `/project/...` path.
- The sandbox blocks directory traversal outside the workdir for safety.

## Typical inspection workflow

1. `./workbench list sessions` to find a session ID.
2. `./workbench show session <id>` or query `workbench.db` to inspect goals.
3. `./workbench list runs <id>` plus `workbench.db` to review a specific run.
4. Tail `<dataDir>/agents/<agentId>/log/events.jsonl` or inspect `/log` via the TUI to see agent/host interactions.
5. Explore `/workspace` (and the run directory on disk) to replay what the agent created or wrote.

## Mounting summary

| Mount        | Host path                          | Description                                                        |
| ------------ | ---------------------------------- | ------------------------------------------------------------------ |
| `/project`   | `<workdir>`                        | Host workspace mounted read/write (configurable with `--workdir`). |
| `/workspace` | `<dataDir>/agents/<agentId>/workspace` | Agent-local workspace for ephemeral files.                           |
| `/log`       | `<dataDir>/agents/<agentId>/log`       | Logs authored by built-in tools and internal agents.                 |
| `/inbox`     | `<dataDir>/agents/<agentId>/inbox`     | Task inbox for the run.                                             |
| `/outbox`    | `<dataDir>/agents/<agentId>/outbox`    | Task results written by the agent.                                  |
| `/plan`      | `<dataDir>/agents/<agentId>/plan`      | Planning workspace (`HEAD.md` + `CHECKLIST.md`).                    |
| `/memory`    | `<dataDir>/memory`                 | Shared memory (`MEMORY.MD`, plus daily `YYYY-MM-DD-memory.md`). |

Keeping this mapping in mind helps when debugging via the CLI or via `ls`/`cat` on the host filesystem.
